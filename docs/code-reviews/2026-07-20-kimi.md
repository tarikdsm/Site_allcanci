## 3. Achados Médios

### 3.1 🟡 Ausência de Content-Security-Policy e meta tags de segurança
**Arquivo:** `src/layouts/Layout.astro:20-34` · **Categoria:** Segurança

GitHub Pages não permite headers HTTP customizados, mas CSP pode ser definida via meta tag — hoje não há nenhuma. O risco de XSS atual é baixo (sem sinks perigosos), mas CSP é defesa em profundidade barata.

**Correção (validar contra o `dist` gerado, pois o Astro inlineia CSS/JS pequenos):**
```html
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; img-src 'self' data:; style-src 'self' 'unsafe-inline'; script-src 'self'; base-uri 'self'; form-action 'none'; upgrade-insecure-requests">
<meta name="referrer" content="strict-origin-when-cross-origin">
```

#### Validação Codex — 2026-07-20

- **Status:** CONFIRMADO
- **Evidência:** no HEAD validado, `src/layouts/Layout.astro` não declarava CSP nem política explícita de referrer, e a resposta publicada pelo GitHub Pages também não continha os headers `Content-Security-Policy` ou `Referrer-Policy`. O build anterior do Astro inlineava o módulo JavaScript no fim de `dist/index.html`; por isso, aplicar literalmente `script-src 'self'` bloquearia o simulador, enquanto acrescentar `'unsafe-inline'` enfraqueceria a defesa. CSP via `<meta>` só governa conteúdo posterior e não suporta Report-Only, `report-uri`, `frame-ancestors` ou `sandbox`; portanto, não foi atribuída a ela proteção de clickjacking nem telemetria que esse modo de publicação não fornece.
- **Ação:** adicionadas em `src/layouts/Layout.astro`, antes de qualquer recurso governado, uma CSP sem `'unsafe-inline'`/`'unsafe-eval'` (`default-src 'self'; base-uri 'self'; form-action 'none'; object-src 'none'; img-src 'self' data:; script-src 'self'; style-src 'self'; upgrade-insecure-requests`) e a meta `referrer=strict-origin-when-cross-origin`. `astro.config.mjs` passou a manter CSS e JavaScript em arquivos externos, e `scripts/check-dist.mjs` ganhou uma verificação de regressão sobre a política e o HTML gerado. `npm run build` e o check focado passaram; em Chromium, a página abriu sem violações no console e o simulador continuou atualizando os resultados.
- **Agente:** finding_3_1_csp

## 4. Achados Baixos

### Segurança

**4.1 🟢 Campo `number` do simulador aceita valores fora de `min`/`max`** — `src/scripts/simulador-bind.js:22`
Atributos `min`/`max` de `<input type=number>` não impedem digitação. Digitar `99999` calcula com 99999 (resultados milionários absurdos) enquanto o slider clampa visualmente em 200 — e o range tem `max=200` enquanto o number tem `max=999` (`index.astro:292-301`). **Não é vulnerabilidade** (saída só via `textContent`; `simular()` trata NaN/negativos; testes cobrem o caso inválido) — é defeito de robustez/consistência.
```js
const clamp = (v) => Math.min(999, Math.max(1, Math.floor(Number(v)) || 1));
const sync = (v) => { const n = clamp(v); faixa.value = n; numero.value = n; render(n); };
```

#### Validação Codex — 2026-07-20

- **Status:** CONFIRMADO — CORRIGIDO.
- **Evidência:** o `range` declarava `max="200"`, enquanto o `number` aceitava até `999`, e `simulador-bind.js` convertia e renderizava todo texto recebido no evento `input`. Assim, underflow, overflow e estados transitórios de digitação podiam deixar controles e resultados divergentes; quantidades finitas extremas também faziam o core produzir `Infinity`.
- **Ação:** definido em `simulador-core.js` um contrato único de 1 a 999 professores, usado pelo core, pelo binding e pelos dois inputs. Valores válidos sincronizam e recalculam imediatamente; vazio, NaN, fração e valores fora da faixa permanecem transitórios durante a digitação e só são restaurados, normalizados ou clampados no evento `change`, evitando que o campo salte enquanto o usuário ainda edita. O core limita quantidades finitas ao mesmo intervalo e mantém resultados inválidos em zero. Os testes cobrem underflow, overflow, NaN/vazio, contrato do markup, sincronização imediata de valores válidos e ausência de `NaN`/`Infinity` nos resultados. `npm run verify` passou com 20 testes, zero diagnósticos do Astro, build concluído e validação do `dist` OK.
- **Agente:** finding_4_1_input_range

#### Validação Codex — 2026-07-20

- **Status:** CONFIRMADO — IMPLEMENTADO.

**4.3 🟢 Clickjacking tecnicamente possível** — a publicação direta no GitHub Pages não envia `X-Frame-Options` nem CSP HTTP com `frame-ancestors`; essa diretiva é ignorada em CSP declarada por `<meta>`. Impacto prático baixo: site sem login, sem sessão, sem formulário e sem ação sensível no servidor.

#### Validação Codex — 2026-07-20

- **Status:** CONFIRMADO — BAIXO; RISCO ACEITO NA HOSPEDAGEM ATUAL.
- **Evidência:** `src/layouts/Layout.astro:38-41` declara a CSP somente por `<meta http-equiv>`, sem `frame-ancestors`. Acrescentar a diretiva ali não resolveria: CSP Level 3 determina que `frame-ancestors` seja ignorado em políticas entregues por `<meta>`, e `default-src` não funciona como fallback para ela. Da mesma forma, `X-Frame-Options` só é efetivo como header HTTP e é ignorado quando simulado por `<meta http-equiv>`. Em 2026-07-20, as respostas `200` de `https://tarikdsm.github.io/Site_allcanci/` e `/index.html` retornaram `Server: GitHub.com`, mas nenhum dos headers `Content-Security-Policy` ou `X-Frame-Options`. O site expõe apenas dois inputs locais do simulador (`src/pages/index.astro:291-306`) e links/CTAs; não há formulário, autenticação, cookie, persistência nem requisição que altere estado no servidor.
- **Ação:** nenhuma alteração de código. Não foi adicionada falsa proteção via meta CSP, meta `X-Frame-Options` ou JavaScript de frame-busting, pois nenhuma dessas opções impede enquadramento de modo confiável nesse modelo de hospedagem.
- **Risco residual / alternativa futura:** uma página maliciosa ainda pode enquadrar e sobrepor a interface para induzir cliques em links, inclusive CTAs externos; sem ações privilegiadas, o dano provável se limita a navegação/social engineering e impacto reputacional. Se o site futuramente ganhar login, formulário ou operações com estado, migrar a entrega — ou colocar um CDN/proxy configurável à frente — e enviar `Content-Security-Policy: frame-ancestors 'none'` como header HTTP (opcionalmente também `X-Frame-Options: DENY` para clientes legados) passa a ser requisito antes da publicação.
- **Agente:** finding_4_3_clickjacking
